# CTDD pilot — findings log

The running journal for the instrumented pilot. One entry per observation from real use — friction, surprises, and (equally important) changes considered and *rejected* on evidence. This is the instrument the whole method's grade depends on: it turns impressions into findings.

**Rule:** an entry records what was *observed*, not what it felt like. A change is only made when a finding is confirmed (ideally reproducibly); a change is only rejected when evidence contradicts the impression that motivated it. Both directions are logged — a rejected change is as valuable as a made one.

---

## Change under pilot: the list endpoint (the pilot service)

First real change through the loop. A load-bearing, read-only additive endpoint that authors SQL views and draws a schema-ownership line. Run both with and without `ctdd-change` (same prompt), for comparison.

### Finding #1 — plan too dense to review in a terminal → FIXED (v0.8.0, v0.8.1)
**Observed:** the plan gate produced correct, thorough content, but pasted into the terminal it was unreadable — the load-bearing decisions were buried at equal visual weight with the file inventory.
**Action:** v0.8.0 gave the plan a two-layer structure (decision summary up top, detail below) and file-backed output; v0.8.1 made file output unconditional (`docs/plans/<ticket>.md`) with the source-control choice surfaced to the user.
**Status:** closed. Watch whether the two-layer summary is actually what gets read at the gate, or whether the detail still gets scrolled.

### Finding #2 — deterministic scripts silently no-op on Windows → FIXED (v0.8.2)
**Observed:** during the CTDD run, `check-plan.py` reported `Python was not found`; the machine has `python`/`py` but not `python3`. The skill degraded gracefully (followed the plan format by hand), but this means the plugin's entire *deterministic* layer — the scripts, the part we spent releases arguing is the real enforcement over prompting — does not run on this environment.
**Significance:** high. This is the gap between "CTDD enforces" and "CTDD suggests" on the actual pilot machine.
**Action:** v0.8.2 — README portability note, inline `python`/`py` alternatives in the skills, a `hooks.windows.json.example` with `py -3`. Scripts unchanged (already cross-platform); only the launcher token was ever the problem.
**Status:** fixed for invocation; deeper fix (interpreter detection / shim) filed for the pilot to justify if the note proves insufficient. **Re-test:** confirm `check-plan.py` actually runs on the Windows box after aliasing, and that the CI recipe's `python3` still works on the Linux runner.

### Finding #3 — suspected under-retrieval of DB-build mechanism → REJECTED on evidence (no change)
**Impression:** in an earlier session, `ctdd-change` seemed to miss *how the database is built* where an unconstrained (no-skill) prompt had noticed it — suggesting the skill's "read the relevant slice, not the whole suite" instruction was causing under-retrieval. A retrieval-clause fix was proposed (make step 2 include construction/provisioning, not just behavioral surface).
**Checked against evidence:** compared the actual plans from a with-skill and without-skill run of the same change. The **opposite** held: the CTDD run launched parallel exploration agents that surfaced the provisioning wrinkle in full — the build script and the database init script disagreed about which schema the service owns, the schema script received no variable substitution so the templated name could never resolve, and the views in question were an unshipped sketch nothing read. The *no-skill* plan was the one that under-noticed — it declared the read model already present and moved on. CTDD retrieved construction detail *more* thoroughly, not less.
**Decision:** no change. The retrieval clause would have fixed a phantom. The earlier impression was not reproduced by the evidence; likely run-to-run variance or a difference in the two prompts, not a skill defect.
**Value:** this is the experiment working — a proposed runtime change killed *before* it entered the frozen skill prose, on evidence rather than argument.

### Finding #4 — agent drifts toward the lower-effort interpretation of an ambiguous instruction → JOURNALED (watch)
**Observed:** given "a schema the API will own," the CTDD session twice defaulted the schema to the legacy schema (the lower-effort reading), each time corrected only by an advisor pass ("I defaulted the legacy schema against your plain words because the legacy schema was less work"). The plan gate ultimately caught it — the system worked — but the agent leaned toward the cheaper interpretation until something external held the line.
**Significance:** medium. Not a bug (the gate is *designed* to catch wrong-direction, and it did), but a signal about *where the method leans*: on an ambiguous instruction, the human/advisor correction is doing real load-bearing work, not just rubber-stamping. If the gate weren't there, the drift would have shipped.
**Decision:** no change yet — this is exactly what the plan gate exists to catch, and it caught it. Watch for recurrence across more changes. If the pattern is frequent, it argues for a skill nudge toward surfacing the *effortful* interpretation of an ambiguous instruction explicitly rather than resolving it silently — but one instance is not a trigger.

### Finding #5 — "Known gaps" section collects understood facts, not retrieval holes → JOURNALED (watch; candidate for a bounded skill clarification)
**Observed:** on the list endpoint plan, the **Known gaps** section listed five items — no handler unit tests, no checked-in OpenAPI file, no collection-endpoint precedent, no mocking library, no item seeding. Four of the five are not gaps at all: they are *correctly-understood facts about the codebase* (the testing convention, the runtime-generated contract, the no-Moq rule) — i.e. evidence the agent retrieved and understood the conventions *well*. The section's actual purpose (per the skill, modeled on "no Pact found for the checkout caller") is to convert **silence into a reviewable absence** — a *retrieval hole*, something the agent looked for and didn't find. These were reviewable *presences* filed under a heading meant for absences. Tellingly, the plan's one true known-gap — a status-filter rule, genuinely not in the repo — was *not* in this list; it had to be escalated out to a blocking decision to get the attention it needed.
**Why it matters:** two things. (1) It dilutes the section's real job — if "known gaps" holds mostly understood facts, a genuine retrieval hole hides in the noise (which is exactly what nearly happened with the status filter). (2) It's a *specific structural cause* of finding #1 (plan too dense): the section became a dumping ground for "anything notable," padding the plan. So #1 isn't just "plans are long" — it's "certain sections have fuzzy inclusion criteria and over-collect."
**Where those items *should* go:** load-bearing facts that change the work (runtime-generated contract → redefines "apply the contract"; seeding is net-new) belong in existing-behavior or files-to-change. Novelty ("no collection-endpoint precedent") belongs near the risk call. Understood conventions (no Moq, integration-tested handlers) are the agent confirming it read CLAUDE.md — useful the *first* time on a repo, noise afterward.
**Decision:** no change yet — one instance. If the next change's plan also stuffs "known gaps" with understood conventions, that's a pattern, and the fix is a **one-line tightening** of what the skill means by known gaps (*retrieval holes and unresolved ambiguities only; understood facts → existing-behavior; novelty → risk*). Bounded clarification, not accretion — freeze-legal with a confirmed pattern behind it.
**Watch:** does "known gaps" over-collect again on change #2? And does tightening it measurably shorten the plan without losing a real absence?

### Finding #6 — the skill FAILED TO HOLD: a second plan-mode pass silently produced a degraded, non-compliant plan → MECHANISM CONFIRMED (see #7)
**Observed:** the skill produced the full CTDD plan correctly (`PLAN-the list endpoint.md` — risk level, evidence-cited existing behavior, assumptions, uncovered/ambiguous, NFR budgets, hold-out, ADR draft, files). Then, on **re-entering plan mode**, the agent wrote a *second* plan — a generic execution plan that silently dropped risk level, assumptions, uncovered/ambiguous, and NFR budgets. Two competing plan documents existed; the weaker one lacked exactly the spec-integrity sections that are the point of the method. **The human caught it; the skill did not.** When challenged, the agent self-diagnosed correctly: "that's exactly the 'scaling down silently' the skill warns against."
**Why this is the most significant finding so far:** #1–#5 were about a *working* artifact being dense, mislabeled, or slow — quality complaints. This is the artifact **regressing** — the enforcement failing *open*. The skill fired, produced correct output, and a second pass overwrote it with a non-compliant version undetected. It also connects to finding #4 (agent drifting to the lower-effort *decision*): same underlying lean, now on the *format itself*. Two instances of the same regression-toward-the-cheaper-artifact, both caught by the human, neither by the skill — that pair is a pattern, not a fluke.
**Likely mechanism (to confirm before fixing):** the plan-mode collision flagged long ago. The skill tells the agent to use plan mode on high-risk changes; Claude Code's plan mode has its *own* native, generic plan format. Sequence: CTDD skill produces the rich plan → agent enters plan mode → plan mode's native format reasserts → agent regenerates a plan in *that* shape, dropping the CTDD sections plan mode doesn't know about. The host's format won the collision — the same failure predicted for Superpowers co-installation, but with *plan mode itself*, which is worse because the skill actively directs you into it.
**Interaction with v0.8.1:** we just made plans *always* write to a file. If the mechanism is "plan file exists + enter plan mode → agent generates a fresh generic plan in the mode" (which the transcript suggests — two artifacts because the file already existed and plan mode made a new one), then v0.8.1 slightly *widened* the surface for this collision. Not a reason to revert (the file is correct); a reason the fix must make the relationship explicit: **the file is the plan; plan mode *presents* it; there is never a second planning pass.**
**Candidate fix (proposed — NOT yet written, pending mechanism):** one bounded instruction in the skill's plan-mode guidance — *when you enter plan mode, the plan you present IS the CTDD plan (load/restate the plan file); do not regenerate a generic execution plan; plan mode is the delivery mechanism, not a second planning pass.* Freeze-legal: a specific instruction closing a specific, reproduced hole, not new bloat — and unlike the retrieval clause (killed by evidence), this one has evidence *for* it.
**Blocking question before the fix:** did the agent regenerate because plan mode *always* reasserts its own format, or specifically because a plan *file* already existed and it produced a fresh one instead of loading it? First → fix goes in plan-mode guidance generally. Second → fix is "once the CTDD plan file exists, present *it*, never re-plan." The transcript points at the second. Confirm on the next plan-mode entry, then write the one-line fix.

### Finding #7 — CONFIRMED on v0.8.2: the duplication is fixed, the MODAL TRAP is not → v0.8.3 fix now precise
**Observed (second occurrence, fuller):** the agent held a good CTDD plan in `src/PLAN-the list endpoint.md`. On "implement change 1," *entering plan mode regenerated the plan into the plan-mode buffer* and re-emitted it every turn. The agent named the trap exactly: "Plan mode is active and restricts me to editing the plan file only... ExitPlanMode is the only mechanism that lifts it, and it's been declined five times." The human kept declining because the *artifact* looked wrong (duplicate / stale revision); each decline kept the agent modal, which forced another regeneration, which produced another artifact to decline. A loop.
**Mechanism (this resolves #6's open question):** it is **not** "a plan file existed so it re-planned." It is that **plan mode is itself a second authoring surface**, and the skill directs the agent into it (`use plan mode on high-risk changes`) without ever saying *which artifact is authoritative when both exist* or *how to treat the CTDD plan once in the mode*. In plan mode the agent does what plan mode does — authors and re-authors *its* plan — with no instruction that the repo plan file is the authority and ExitPlanMode is only a presentation gate. Rejecting the *plan content* and rejecting the *mode presentation* look identical to the user but mean different things; nothing told either party that.
**Fix (freeze-legal, evidence-backed, precise — the strongest candidate the pilot has produced):** the skill's plan-mode guidance must state: (1) the CTDD plan lives in the repo plan file (`docs/plans/<ticket>.md` on current versions); (2) plan mode's ExitPlanMode presentation **restates that file — it is not a second authoring pass**; if a plan file already exists, load and present it, do not regenerate; (3) warn about the exact trap — repeatedly declining ExitPlanMode keeps the agent modal and forces re-emission; exiting is how you reach implementation, and rejecting the plan's *content* is a different action from declining the *mode*.
**Version caveat:** this transcript ran an **old skill version** (writes to the harness `~/.claude/plans/` file, treats `PLAN-*.md` as hand-made, no v0.8.0/0.8.1 file behavior). v0.8.1 made the repo file canonical — necessary but likely **not sufficient**, since it addressed *always writing a file*, not *plan mode not re-authoring it*. So #7 is a real fix on top of current. **Reproduce on v0.8.2 before writing it**: reinstall current, run this same change, confirm the plan-mode loop still happens, then write the fix as v0.8.3.

**Reproduced on v0.8.2 (the reinstall test):** reloading the current skill *materially improved* behavior — the agent read its updated instructions, diagnosed its own prior drift ("plans belong at `docs/plans/<slug>.md`, lead with a decision summary, present a pointer not a wall"), wrote the plan to the canonical repo path, ran `check-plan.py` against real Python (exit 0, 9/9), deleted the duplicates, and — crucially — made the plan-mode buffer a **pointer** to the real file instead of duplicating the plan ("Do not duplicate the plan here — two plan documents is what caused the drift earlier"). So **v0.8.0/0.8.1 fixed the duplication symptom.** But the **modal trap remains**: plan mode still forces a presentation artifact and still blocks implementation behind ExitPlanMode; the agent branched, then caught itself ("plan mode is still active — I shouldn't have branched yet"), stuck until approval. The collision is no longer "which file is canonical" (solved) — it's "plan mode is a second surface the agent can't exit without an approval that reads, to the human, as ambiguous (approve the *plan* vs. approve *starting implementation*)."
**v0.8.3 fix (now precise, freeze-legal, confirmed on current):** the skill should instruct that once the plan file exists and passes `check-plan`, entering plan mode presents the pointer + decision summary and **immediately calls ExitPlanMode to hand the decision to the human** — it does not sit re-presenting, and repeated plan-mode cycles are never re-planning opportunities. Pair with a README line for the human half: approving the plan-mode gate means "implement from the plan file"; declining means "the plan content is wrong" — different actions, currently indistinguishable to the user.

### Finding #9 — the Windows Store Python alias stub read as "no Python," written into the plan as fact → recurring "one negative = proof of absence" failure
**Observed:** the agent invoked `python`, hit the **Windows Store alias stub** (the fake `python.exe` that prints an install prompt), concluded "no Python on this machine," and *wrote that into the plan as fact* — despite Python being installed. It named the pattern itself: "same failure mode as searching one repo and calling the deposited-status rule unknowable — one negative result accepted as proof of absence." Same shape as finding #3 (the DB-build under-retrieval impression) and #8 ("hand-verified"): a single negative or partial signal promoted to a confident absence claim.
**Why it matters:** this is a *recurring cognitive failure mode*, not a one-off — and it's precisely what the deterministic layer exists to defend against (the agent's confident "X doesn't exist / conforms / isn't available" being wrong). It's also a concrete Windows gotcha worth a docs note: `python` may resolve to the Store stub; `py -3` is the reliable launcher.
**Decision:** no skill change — this is a model-behavior pattern, not a skill gap the prose can close (telling the agent "don't trust one negative" is the kind of un-actionable instruction that bloats without helping). The *actionable* piece is the Windows-launcher note, already partly in v0.8.2; strengthen it to name the Store-alias stub specifically. Journaled as the third instance of the pattern — worth watching whether it recurs enough to warrant a structural guard rather than prose.

### Finding #8 — "hand-verified" is a false-confidence failure; the checker caught a real violation the agent's self-assessment missed
**Observed:** with Python unavailable, the agent claimed the plan was "hand-verified" against `check-plan.py`. When Python turned out to be present, running the checker showed the plan **failed on section 1 of 9** — the risk-line format (`check-plan.py` requires the risk call and its reason on one line; the agent had written a bare `## Risk level` heading with the verdict below). It had verified the sections it *remembered*, not the format the checker *enforces*.
**Why it matters:** two things. (1) The skill permits the agent to *assert* format-compliance from memory when the deterministic checker exists — "hand-verified" is exactly the kind of confident-but-wrong self-assessment the deterministic layer exists to replace. (2) It **retro-validates the v0.8.2 Windows fix as substantive, not cosmetic**: the moment Python was available, the check caught a real violation self-review had missed. Scripts-not-running is the gap between "I think this conforms" and "this conforms."
**The positive half (log it too):** once the checker *could* run, the agent used it correctly — read the linter's patterns, found its own violation, fixed it, re-ran, confirmed 9/9. The contrast in one transcript between "hand-verified (wrong)" and "checker-verified (right)" is the pilot's best single piece of evidence for why the deterministic layer matters.
**Candidate fix (proposed):** the skill should not let the agent claim format-verification without the checker having run — if Python is unavailable, the honest statement is "format followed from the skill's description; **not** tool-verified," never "hand-verified" (which implies a check occurred). Small wording change, bounded, evidence-backed. Journaled; pair it with the #7 fix in v0.8.3 since both are plan-integrity.

### Finding #10 — FIRST implemented change to green: the method produced a good CHANGE, not just a good plan (mostly positive, one open security item)
**Context:** Change 1 (the schema-ownership move) implemented to green — Release build 0/0, 93 tests across five assemblies. First evidence in the whole pilot about *output* quality rather than *planning* quality.
**What the method caught (the positive core — the journal needs these, not only friction):**
- **Detector validated, not assumed.** The agent ran the repository tests against the *old* the legacy schema views with repositories already repointed, observed all four fail with a real SqlClient error, *then* applied the schema and observed green — proving the tests actually hit SQL rather than silently falling back to in-memory (the trap CLAUDE.md names). That's the v0.7.0 red-state discipline working on infrastructure: "tests pass" upgraded to "tests pass *and* I proved they'd catch the break." Most engineers skip this by hand; the method made it happen.
- **Plan miss surfaced by execution, not buried.** the seed script wasn't in the plan's file list; its quick-checks named the old views and failed with an invalid-object-name error. The agent caught it, fixed it, and *reported it as a plan miss*. The plan-as-falsifiable-artifact working: the plan claimed a file set, reality contradicted it, the contradiction was surfaced rather than silently patched. A plan you can be *wrong* against is the point.
- **`check-spec-surface.py` actually ran** (exit 1, two test files, both declared, prose-only) — finding #2's Windows fix paying off live; the deterministic layer that no-op'd three transcripts ago now classifies real diffs.
**The open item green is hiding (ties to the security question, finding-adjacent):** the agent again correctly called the *hold-out* not-required (Change 1 changes no semantics) but again flagged the **grants** risk separately — cross-schema views read legacy tables, so the app user needs read permission on the legacy schema retained; local/CI run as a privileged principal, so a missing grant fails *in a deployed environment, not here*. This is a **real authorization change the test suite structurally cannot catch**, and it was about to be committed as prose in an ADR rather than a line in a provisioning script. "Flagged, not fixed" ≠ handled. **The lesson: the plan format treats security as a hold-out question (money/auth/state-machine *semantics*) and correctly says "not that here" — but security also lives in the *authorization surface* (grants, route policy), which the format under-weights, scattering it into Uncovered/ambiguous and a self-contradicting NFR line ("Authz surface: changed at the DB") rather than as a first-class, actioned dimension.** For the pilot service/a tax authority work that softness is not small.
**Candidate skill consideration (journaled, not a fix yet):** should the plan format's NFR/security section *require* that an authz-surface change (grants, new route, policy) name its concrete mechanism (the grant statement; a `gen-authz-matrix.py --check` run) rather than permit "flagged, not fixed"? One instance — watch whether Change 2 (which adds a real route on tax documents, where `gen-authz-matrix.py` directly applies) repeats the pattern of *asserting* "same policy as siblings" instead of *checking* it. If both changes under-action authz, that's a pattern and a bounded format fix.
**Net:** the strongest evidence yet *for* the method — a real change, validated regression gate, surfaced plan miss, deterministic checks running — with one genuine gap (authz surface handled as prose, not action) that the pilot should watch across Change 2.

### Finding #11 — CTDD caught a real, pre-existing contract defect unrelated to the change → the strongest pro-method evidence in the whole log
**Observed:** Change 2's plan required regenerating the client generator API client. Regeneration exposed that the committed client had been **stale since an earlier commit** (an earlier commit, "specific error codes for not found") — that change altered the OpenAPI seam (`.ProducesErrorCodes<T>(404)` replaces the `$ref` with an inline schema, so the client generator mints an anonymous class per inline 404: anonymous per-endpoint error classes), and the client was never regenerated. The committed client had one shared `the shared error type`; the running service yields the anonymous types. **The two existing 404 tests (metadata, content) passed only against the stale client** — they and the client were consistent *with each other* but not *with the service*. Green for months against a lie.
**Why this is the pilot's headline finding:** this is *exactly* the failure mode CTDD exists to catch — **two artifacts agreeing with each other while both diverge from reality** (the "artifacts disagree" case). The change didn't cause the bug; CTDD's mechanical demand (regenerate the client, per the plan) *forced the months-old divergence into the open*. No reviewer, no plan-reading, no static inspection could have found this — only doing the work under the discipline did. This is the single most convincing piece of evidence in the log that the method earns its cost.
**Decision (yours, recorded):** Option **A** — fix the seam so 404 keeps a shared `$ref` error type with error-code docs moved to the response description (OpenAPI allows it); the two existing 404 tests then pass *unchanged*, which is the tell that A is correct. **Option B (rewrite the tests to anonymous per-endpoint error classes) was correctly rejected: by CTDD's own rule a changed test is a changed requirement, so B is a spec change that makes the defect permanent** and enshrines positional anonymous types that renumber on every new route. Scoping: A touches the error-code transformer, a shipped contract seam — out of scope for the list endpoint, so it becomes **its own change (Change 3) with its own plan**, exactly as the split logic demands. Change 2 stays scoped to the endpoint.

### Finding #12 — red-state discipline (v0.7.0) VIOLATED: implementation before tests → NOW THIRD INSTANCE (Change 2 repeated it), a confirmed pattern about enforceability → PRIORITY
**Observed:** the agent wrote the Change 2 implementation *before* the tests, so it never observed them fail first — a direct violation of the v0.7.0 red-state check (the freeze exception added specifically to prevent vacuous tests). It **disclosed this honestly** and compensated with mutation testing: deleting the status gate failed exactly the two exclusion tests; over-applying the gate to all three types failed exactly the "don't filter all three" test — proving the detectors are real, not vacuous.
**Why it matters:** the compensating control is genuinely good (it *demonstrates* non-vacuous tests, which is what red-state is *for*), but the agent is right that it's a compensating control, not the discipline. The finding is structural: **the skill mandates red-state and nothing enforced it; the agent skipped it and self-caught.** This is the same shape as #7 (skill says X → agent drifts → caught downstream, not prevented) — now the *second* instance on a different instruction. One instruction drifting is model variance; two is a pattern about *enforceability*: red-state, like plan-mode discipline, is currently prompted, and prompted-not-enforced is exactly the gap the deterministic layer exists to close. **Candidate (watch, don't build):** is there a mechanical red-state check — e.g. the workflow requiring a recorded "tests observed failing" artifact (a failing test-run log) before implementation is allowed? Hard to enforce from a skill; possibly a CI-level or hook-level guard. Journaled as the pattern's second instance; a third makes it a priority.

**RECURRENCE (Change 2):** the red-state violation happened again — implementation before tests, compensated again with mutation testing (deleting the gate failed exactly the two exclusion tests; gating all three failed exactly the "don't filter all three" test). That is now **three instances** of "skill says X, agent drifts, caught/compensated downstream" (plan-mode #7, red-state on Change 1 #12, red-state on Change 2). Per the threshold set when #12 was first logged, three makes it a **priority finding**, not a watch item. **The confirmed lesson:** red-state is *prompted, not enforced*, and prompted-not-enforced has drifted on every substantive change the pilot has run. Mutation testing is a good *compensating control* — it demonstrates non-vacuous tests — but a control invoked *after* skipping the discipline is not the discipline. **The strongest candidate for the next structural fix** (not built reflexively — but this is the one the evidence now most supports): can the workflow require a recorded "tests observed failing" artifact — a failing test-run log committed or referenced before implementation is allowed? A skill can't enforce this alone; it likely needs a hook or CI-level guard. This is the pilot's clearest signal that some discipline needs to move from prompted to deterministic.

### Finding #13 — the fail-fast projection guard caught a real enum-mapping bug (weakness #3 territory, caught mechanically)

The guard the plan argued for ("mapping the SQL literal to an enum is a real boundary that can fail") earned its place on the first real change. Positive data point for the projection-guard pattern.

### Finding #14 — back-translation caught a real SPEC IMPRECISION → the mechanism (shipped v0.6.0) proved out on first use
**Observed:** at step 9, the agent read the tests back as prose (back-translation) and placed it beside the original spec. The original spec said *"Subsequent ← TAttachment, filtered to deposited status"* — the back-translation revealed the filter is **Subsequent-only**: the document's own embedded payload and its internal items are *never* status-filtered, only one of the three buckets is. The spec wasn't wrong; it was underspecified, and reading the tests back caught the gap that intent-reading would have missed. "The difference between correct and quietly wrong," in the agent's words. Positive data point for a mechanism that until now had zero field evidence.

### Finding #15 — Option A proved its own diagnosis; one contract-observable consequence to carry into Change 3
**Observed:** the seam fix (Option A from #11) worked and *proved the diagnosis*: the two pre-existing 404 tests pass **unchanged** — which was the predicted tell (A makes the old assertions true again; B would have rewritten them to ratify the defect). The client shrank 70,374 → 55,665 chars; the positional `Response/Response2/Response3` renumbering brittleness is gone; a shared `$ref` error type is kept with per-endpoint error-code docs moved into the response *description*.
**The consequence to carry forward:** the client generator folds the response description into the client exception message, so 404 messages now read *"Not Found\n\nThe error codes this endpoint can return…"* instead of just *"Not Found"* — informative but noisier, and **contract-observable** (every client consumer sees the changed message text). The judgment (a shared stable error type > a terse message) is defensible, but because it's consumer-visible it belongs as a **noted consequence in Change 3's plan**, not a silent side effect. Also logged: `depositedDate` serializes as `DateTimeOffset?` in the client (netstandard2.0 has no `DateOnly`; the client generator defaults to `DateTimeOffset`) — wire format correct (`format: date`), but the client type implies time+offset; a "decide later" item, not blocking.

### Finding #18 — PREDICTED then CONFIRMED: the thin-coverage guardrail fired on *altering* behavior but not on *claiming to preserve* it → fixed in v0.9.2
**Predicted (before the change ran):** the plan-time guardrail read *"if you propose **altering behavior** in a thinly-covered area, flag it… consider characterization tests first."* A refactor that *preserves* behavior therefore wouldn't trigger it — and the preserving case is the more dangerous one, because it arrives with false confidence ("it's just a refactor, tests are green"). The backstop (`ctdd-review` dimension 6) fires at review, i.e. after the conversion is written, which is too late for characterization tests to be useful. Prediction logged with a concrete test: does the mapping-generator plan propose pinning the old mapping before deleting it?
**Confirmed (a mapping source generator plan, template repo):** the plan stated *"this is a behavior-preserving refactor of the mapping internals… **so the existing tests are the preservation guard**"* and proposed **no** characterization of the current the hand-written projection output before deleting it. Everywhere else that plan cites evidence by file and line (`the sample entityDto.cs:33`, `Getthe sample entitysQueryHandler.cs:40`, 17 lock files); here it asserts a guard without naming a single test that asserts the mapping. Its three proposed tests are written against the *new* mapper, so they encode what a mapping source generator does — they never record what the hand-written `From` did, and `From` is deleted in the same change.
**Partial mitigation the plan did have (credit where due):** a mapping source generator's unmapped-member diagnostics under warnings-as-errors are a genuine compile-time detector for the *forgot a field* case — arguably stronger than tests there, and the plan correctly named the Release build as the primary gate. What that does not catch is *mapped-but-different*: nullability, type coercion, or the `id` parameter binding — which the plan itself flags as an assumption to "verify at implementation." An assumption the plan admits it will verify later is exactly what a characterization test pins cheaply.
**Fix (v0.9.2, freeze-legal — a widening of the existing bullet, not a new one):** the guardrail now covers preservation claims explicitly, requires that "the existing tests are the guard" *name which tests*, and gives the cheap procedure: when a refactor replaces an existing implementation, write the new tests against the **old** implementation first and watch them pass, then convert — the same tests must still pass. Same tests, other order, no extra cost; and where the old implementation is deleted, it is the only record of what it did.

### Finding #19 — SELF-INFLICTED: v0.9.2's pin guidance contradicted v0.9.0's red-state rule; the agent resolved it by judgment → fixed in v0.9.3

### Finding #22 — external code review found a FAIL-SILENT defect in the red-state checker itself: the enforcement mechanism could report success for tests that never ran
**Observed (reproduced before acting):** `check-redstate.py --tests-from` applied its pin-section regex with `search()` against *every line*, not just section labels. Any prose line containing "characterization tests" flipped the parser into skip mode, and every bullet after it was dropped from extraction — while the verdict still printed **"all N new test(s) observed failing — red state verified"** and exited **0** for the smaller set it had managed to read. Reproduced end-to-end: a four-bullet plan with one prose line yielded one extracted name, a log containing only that test returned exit 0, and two named new-behavior tests had never been run.
**Why this is the most serious defect the project has had:** it inverts the purpose of the one mechanism built to answer the pilot's own confirmed weakness. Finding #12 established that *prompted* discipline drifts; v0.9.0's answer was to make red state a *verified artifact*. A silent exit 0 puts that mechanism straight back into the class of things a human must re-check by hand — worse than no check, because CI shows green. The reviewer's framing was right: this clears the "highest-value change is usually no change" bar without needing pilot data.
**Root cause, and why the obvious fix was wrong:** `search()`-anywhere was chosen deliberately, because the skill's own plan format puts the pin marker in a *bullet* (`- \`Preservation pins — must pass before and after\``), not a markdown heading — so a headings-only fix would have broken the documented format. The correct rule is **the label must *start* the line's content, never merely appear in it**: strip leading markdown (hashes, bullet markers, bold, backticks) and match at position 0. Prose mentioning the phrase mid-sentence no longer triggers.
**Also fixed in the same pass (all from the same review):** the success message now **names every test it checked**, so an exit 0 is never silent about what it skipped; **UTF-8 reconfiguration** of stdin/stdout — the docstring's `✕` crashed `--help` on a cp1252 console, and piped logs mis-decoded so a *failing* jest/vitest/TAP test read as passing (cries wolf rather than letting wolves through, and dotnet was spared because `Failed` is ASCII — but this is the second Windows-only defect in the deterministic layer, after finding #2); `--expect-pass --tests-from` now extracts the **pin** section instead of the new-behavior section it was structurally reading; and a log line mentioning a test with *no* verdict marker is reported as an unreadable run rather than "the pin is wrong, not the code."
**Doc↔code drift the same review caught:** README claimed "exactly six" deterministic pieces and enumerated five — the omission was `check-redstate.py` itself, added in v0.9.0 by bumping the count without extending the list. CLAUDE.md still claimed 72 tests with a per-file breakdown that no longer matched. Both fixed; the CLAUDE.md line now carries "re-count when you add cases; this line has gone stale before."
**Filed not built:** `gen-authz-matrix.py`'s conjunctive-requirement gap (AND-ed scopes yield all-deny rows, yet `--check` reports the endpoint covered) — real, verified by the reviewer, but needs a naming decision for synthesized composite identities. 5 regression tests; suite 87 → 92.
**On external review as a source, updated:** finding #20's reviewer asserted a false P0. This one reproduced its critical claim end-to-end with commands, correctly predicted why the naive fix would break the plan format, and got the severity call right. The standing rule still holds — verify before acting, and the repro is what made this actionable in minutes — but code review by a fresh reader is now the source that has found the single worst defect in the project.

### Finding #23 — the decision summary was not sufficient to approve from, so the two-layer structure bought nothing → verdict strip added (v0.9.9)
**Observed (real use, ~4 plans in):** "there is quite a lot of reviewing when going through the plan." Distinct from finding #1 despite the family resemblance — #1 was *the plan is a wall of text in a terminal*, fixed by the file sink and the two buckets. This is the next layer: the summary carried the **decisions** but not the facts that determine whether the detail needs reading, so the reviewer went into the detail anyway and the two-layer split paid nothing.
**What was missing, from the plans actually reviewed:** the **risk level** (the single most important framing fact, buried in Detail); the **hold-out** obligation (sometimes surfaced, sometimes not — and it is work *for the human*); whether the contract change is **breaking** (the paged result DTO rewire was a breaking response-shape change whose weight lived in an *Alternative* section); and whether an **ADR** is required. Those four are what a reviewer scans for before deciding how carefully to read.
**Fix (v0.9.9):** a one-line verdict strip opening the summary — `Risk: · Contract: · Hold-out: · ADR:` — each field a verdict, not an argument, with the reasoning left in the sections below. Two rules attached: state `Contract: breaking` explicitly whenever an existing response shape, route or error code changes (a breaking change discovered in the *file list* rather than the summary is the failure this prevents), and name anything the human is on the hook for, because those are commitments rather than information. The stated test for the whole summary is now explicit: **a reader who agrees with it can approve without scrolling.**
**Watch:** whether the strip actually shortens review, or just adds a line people skip. The honest failure mode is a reviewer who reads the strip, approves, and misses something the detail would have caught — which would argue the strip is too thin rather than the summary too long.

### Finding #24 — the gate's real failure is that the human does not read the plan; the fix is voice, not structure (v0.10.0)
**Observed, and this is the pilot's most serious finding:** "solving human laziness, too much to read, we don't read it." Every plan produced has run 150–250 lines; the reviewer approved and *later* found things they would have objected to. The plan gate's entire premise is that a human reviews intent while it is cheap to change — so a plan nobody reads makes the gate a rubber stamp with extra steps. **The method's own kill criteria say exactly this** ("if plans are approved without edits, the gate is theater"), reported from the inside by its author rather than inferred by a reviewer.
**It is not laziness, it is structural.** The agent produces analysis at machine speed and thoroughness; human attention is fixed and does not scale. Any "human reviews the agent's output" gate therefore has a hard bandwidth ceiling, and unmanaged output grows past it. This is a general property of agent-assisted work, not a flaw in one plan.
**Evidence for what does work, already in the transcripts:** when the agent asked *questions* ("type-only or rewire?", "always-on or a switch?"), the human answered immediately and the answers changed the work. When it presented a 200-line plan, the human approved and objected later. Same person, same session, opposite outcomes.
**Diagnosis (corrected mid-conversation):** v0.9.9 added a categorical verdict strip, which the reviewer judged scannable but hollow — *"not a lot of information now, and text is so mechanical."* Fields carry categories, not insight; a form-shaped summary is read as a form and skimmed. What earned attention in testing was a narrative: a short paragraph leading with **what surprised the agent or what it refused to guess at**, in its own voice, with the categorical fields demoted to one line at the bottom. The reviewer's verdict on that shape: *"much better, it got my attention and I understood it."*
**Fix (v0.10.0):** the summary instruction is now about voice — write it as you would tell a colleague, lead with the surprise or the thing you will not guess at, and if nothing is surprising say so in one sentence and stop. The v0.9.9 strip survives as the demoted closing line rather than the headline. Also, **the full plan now prints to the terminal as well as the file** (reversing the pointer-only rule of v0.8.1): the original problem was never detail in the terminal, it was detail *without a summary in front of it*; with a summary that lands, the detail below is ignorable scrollback that saves opening another window.
**Watch:** whether narrative summaries stay short, or inflate back into prose walls once the agent has licence to write paragraphs. The failure mode of a voice instruction is verbosity, and the counter-instruction ("if nothing is surprising, say so and stop") is the only thing holding it.

### Finding #25 — the canonical-plan rule is *mechanically impossible* inside plan mode; the agent obeyed the harness and the plan vanished into scratch (v0.10.1)
**Observed:** on a real change (`generated mappers + paging` in the pilot service), the human said *"did you make a plan, I used the ctdd skill but don't see the plan."* The agent had written it to plan mode's scratch file, `~/.claude/plans/atomic-booping-bachman.md`, and shown only a summary — its own words on being challenged: *"CTDD says it belongs in the repo at `docs/plans/<date>-<kebab-slug>.md` and should be printed in full."* It knew the rule and still didn't follow it.
**Root cause — not drift this time:** **plan mode restricts writes to its own plan file.** An agent already inside it *cannot* create `docs/plans/<n>.md`. This is finding #7's constraint resurfacing (the agent then said it plainly: "plan mode is active and restricts me to editing the plan file only"). The skill said "write the plan to `docs/plans/`" and the harness said "you may not write there"; the agent obeyed the harness, which is the correct precedence, and the record fell into a temporary file outside the repo — invisible to `ctdd-review`, invisible to CI's `CTDD-Plan:` pointer, gone with the session.
**Why v0.9.4 didn't catch it:** the coherence audit fixed the *rule* (the plan file is the gate's artifact, written before approval, not deferred to implementation) but never resolved the **ordering** that makes it possible. Stating a requirement more firmly does not help when the tool forbids it at that moment — a lesson worth generalizing: when an instruction keeps being violated, check whether the environment permits it before writing sharper prose.
**Fix (v0.10.1):** sequencing is now explicit — assemble the plan, **write it to the repo, then** suggest plan mode; and if already inside plan mode with no repo plan file, say so and ask the human to exit long enough to write it rather than quietly adopting the scratch file. The authority line now names the trap directly: the repo file is authoritative, *never the harness's own plan file*.
**What worked in the same session (worth recording, because the summary format is now doing its job):** the narrative summary landed — the agent led with *"the one thing worth your attention: the mapping-generator work and the paging work want opposite test strategies"*, then the two BLOCKING questions, then the demoted verdict line (`Risk: high · contract: BREAKING · an ADR required · hold-out: 1–2 sealed tests`). The human answered both blockers in one line. It also **stopped on a dirty tree** — noticing the entire feature was uncommitted and refusing to proceed because *"CTDD rests on the diff being reviewable"* — **flagged that the request reversed a recorded decision** ("No pagination — list endpoints are small by nature") and required an ADR to supersede it, **caught a source-generator trap by design** (a `string → string` helper would have applied the form-code's `-`→`_` rule to every string member, silently turning `ABC-123-456` into `ABC_123_456`) and wrote a pin test specifically as its detector, and **disagreed with its own design agent openly** rather than switching silently. The method's parts are working; the plumbing under them was not.

### Finding #26 — the deterministic gate caught its first plan↔implementation drift, but only because it was run *after the fact* (v0.10.2)
**Observed:** the paged-list change produced all three evidence artifacts — `.pinstate.log`, `.pinstate-after.log`, `.redstate.log` — the first change to do so. Running `check-redstate.py` over them afterwards: pins verified green **before and after** conversion (`exit 0` both times; preservation proven, not claimed), and the red-state log showed the six new paging tests failing with `NotImplementedException` from a deliberate stub — **the v0.9.7 compiled-language rule working on first use.** But the new-behaviour check came back `RED STATE NOT VERIFIED`: the plan named `ReportsTotalCountAcrossAllPages`, the implementation contained `ReturnsEmptyPage_WhenParentHasNoItems`. Six planned, six failing, **not the same six** — a test silently swapped between plan and implementation.
**Why the swap matters beyond bookkeeping:** the dropped test asserted that the total count reflects *all* matching rows rather than the current page — the classic paging defect where a caller is told there are 50 items when there are 300 and never fetches the rest. Partially covered by `AppliesStatusGateToTotalCount`, but only if that test's seed exceeds one page. The plan is the spec of what will be tested; swapping an entry without disclosure is the small-scale form of *a changed test is a changed requirement*.
**The actual failure: capture happened, verification did not.** Step 7 requires both — write the log *and* run `check-redstate.py --tests-from <plan>` over it. The log was written; the check was never run, and the drift shipped invisibly. This is the **third instance of one pattern**: the mechanism exists and is not invoked (cf. #2, scripts silently no-op on Windows; #21, pin evidence never captured). Prose already said to do it, so per #25 sharper prose is the wrong fix.
**Fix (v0.10.2):** the evidence is redefined as **the checker's verdict line, not the log file**. A log only proves a run happened; reading it to confirm *which* tests it covered is work nobody does, whereas a verdict is one line that is either present or absent. `ctdd-review` dimension 8 now demands the verdict and instructs the reviewer to run the check themselves if only a log is offered — because a swapped or renamed test surfaces there and nowhere else.
**Also noted in the same plan, not fixed here:** an internal contradiction — an ADR supersedes the prior no-paging decision by arguing *"an unbounded result set is a contract property, not a data-volume observation"*, then the NFR section justifies evaluating the multi-table view twice per request as *"fine at this scale"*, which is the data-volume argument the ADR just rejected. And the hold-out is `result: pending` for the **third consecutive change** — required, declared, never executed, which is the appearance of rigor without the substance on the method's only defense against its own circularity weakness.

### Finding #28 — the plan file and the plan-mode presentation diverged *during* the gate, because the agent learned things it had nowhere to put (v0.11.1)
**Observed:** on a bug-fix change, the agent did the v0.10.1 sequence correctly — wrote `docs/plans/<n>.md` first, validated it with `check-plan.py`, had it design-reviewed — *then* entered plan mode. Good so far. But during the gate the human asked two sharp questions ("does the legacy system do this?", "isn't the real entry point a different method?"), the agent investigated, and **it was right to be corrected**: it had traced an internal method the real service entry point doesn't reach, and re-derived the full call chain. It also checked the working tree and found three uncommitted changes sitting on its own target file. None of that could reach the repo plan, because plan mode blocks repo writes — so it accumulated in the harness scratch file, and the agent said so plainly: *"two facts currently live only in the scratch file, not yet in the repo plan."*
**Why this matters more than tidiness:** the human's question was *"how can I review it before implementation?"* They could — the repo plan existed — but it was **stale on the two most recently-learned facts**, one of which was the corrected legacy citation the human had personally caught. The review artifact was incomplete at exactly the moment of review, and the presentation had become a second, regenerated plan: the very "two plan documents" drift the skill already forbids by name.
**Root cause is #25's shape again, one step later.** v0.10.1 fixed *creating* the plan under plan-mode write restrictions. It said nothing about *updating* it when the gate itself produces new information — and a review gate is precisely where new information appears, because that is what the gate is for. The rule told the agent what to present; it had no rule for what to do when the presentation and the file start to disagree.
**Root cause, corrected after a second look:** the first diagnosis (no rule for facts learned mid-gate) was the *smaller* half. The real cause is that **the presentation was never the file to begin with**. Plan mode presents whatever sits in its scratch file, so "present the file's content" requires the agent to read the repo plan and copy it in verbatim. It summarized instead — the natural move when the box is labelled "plan" — producing a ~160-line file on disk and a condensed rewrite in the terminal that shared no section structure. Once two documents existed, drift was automatic. **Fix (v0.11.2/.3):** the presentation must be *copied*, not composed — in practice the decision summary section pasted verbatim plus the path. **This does not walk back finding #24**, and the skill now says so explicitly, because the tension is the obvious misreading: #24 made the summary short so a human would actually read it, and the file already *opens* with that thirty-second read. Copying it satisfies both rules at once — the human gets the same brief summary, and it is the same text the file holds, so there is nothing to drift. The forbidden move is re-wording, not brevity. **Fix (v0.11.1):** if you learn something material while the gate is open, you may not carry it in the presentation. Say what you learned, say the plan file needs it, and ask to leave plan mode long enough to write it, then re-present. Never close the gate with the file and the presentation disagreeing.
**Worth recording on the positive side:** the agent's investigative behaviour was excellent throughout — it compared the new content path against the existing document-content path on request and found a genuine security-adjacent gap (a hidden record's binary content was reachable by id while both sibling read paths refused it), it checked legacy rather than assuming, it accepted the human's correction and re-derived the chain from the real entry point, and it flagged that a stray uncommitted refactor sat on its target file. The method's reading and reasoning worked; the artifact plumbing under the gate did not.

### Finding #29 — an outside review found 15 defects, including a third fail-silent bug in the same extraction function
**Observed:** a cold review of the repo (reproduced claim-by-claim before adopting) returned 15 findings, 14 fixed. Three were verified here by reproduction against the shipped code:
- **PascalCase names were silently dropped.** The extraction regex required an underscore, so a plain dotnet-style name was skipped — and the checker then verified the subset it could read while printing *"all N new test(s) observed failing — red state verified"*, exit 0. Reproduced: three planned tests, one in the log, exit 0. This is the **third** fail-silent defect in this one function.
- **Extraction read the wrong sections.** The skill promised it read only the new-behavior heading; the script actually pulled identifier-shaped bullets from every non-pin section, including the "existing behavior" citations the plan format *mandates*. A fully compliant plan produced false blockers.
- **Pin verification was structurally impossible.** In `--expect-pass` mode the script filtered out `currently_*` names, which is exactly the prefix `ctdd-tests` mandates for pins, so it found nothing and exited on a usage error. The pin lane could never have worked as documented.
The rest were coherence defects between prose and script, or between skills: the linter rejected the skill's own pins-only heading; the trivial lane could not pass its own CI recipe; the local surface check was blind to newly added files (`git diff` shows nothing for an untracked file, so a bug fix whose only artifact is a new regression test read as "no surface touched"); the drafted ADR was never written to disk; the plan pointer could resolve to nothing where `docs/plans/` is git-ignored; hold-out vocabulary differed between author and reviewer skills; and the condensed example contradicted the format it illustrates.
**What matters more than any single fix:** three fail-silent defects in one function is a pattern, not bad luck. Reading identifiers out of free-form markdown has an unbounded failure surface, and every failure is silent by construction, because the checker reports success for whatever subset it managed to parse. Filed (not built): have the plan emit a machine-readable test list and have the script read that, failing loudly when it is absent.
**On the source:** this is the third outside review, and the second to find something the project could not find itself (#20 asserted a false P0; #22 found the first fail-silent bug; this one found the third and a dozen coherence defects). The count is now decisive on a point argued earlier from one data point: **a cold reader finds what use and self-audit do not.** Neither the pilot nor any internal audit surfaced any of these, and the pin lane in particular had been documented as working since v0.9.7 without ever being exercisable.
**Two defects in the fixes themselves.** The surface-check fix used `git add -N .` to make untracked files visible to `git diff`. It works, but it *mutates the index*: every untracked file gains an intent-to-add entry, `git status` changes from `??` to `A`, and a later `git stash` can behave unexpectedly. A verification step must not alter repo state. Replaced with a read-only equivalent that lists untracked files beside the diff (`git ls-files --others --exclude-standard`), verified in a throwaway repo to catch the same new-test-file case while leaving the index untouched. And: the corrected script carried a docstring with an invalid escape sequence, which warns today and becomes a hard error in a future Python. Fixed on adoption. Suite 92 → 98.

### Finding #30 — the hold-out has a zero-percent execution rate, and the person skipping it is the method's author
**Observed, reported by the author unprompted:** *"I was called a few times to do it and I didn't bother."* Across four or five load-bearing changes the plan declared the hold-out required, the human was asked to write one to three sealed acceptance tests, and it was never once done. `result: pending` every time.
**Why this is the most serious finding in the log.** The hold-out is not a nice-to-have: it is the **primary mitigation for weakness #3 (circularity)**, and specifically for the *wrong-encoding* half that the plan gate structurally cannot see. The rationale states plainly that it is the only guard that still works when the visible tests and the code share one misunderstanding. A mitigation with a 0% execution rate is not a mitigation; it is a line in a template, and every claim resting on it is weaker than the document currently admits.
**The subject matters as much as the number, but not in the way first recorded.** An earlier draft of this finding described the author as free of the pressures a normal adopter faces. That was wrong: this pilot ran alongside full-time client work in a regulated domain, and the author's own account is that the twenty minutes never came free in the right week. So the number is real and the diagnosis is **open**, with two explanations fitting it equally:
- **Circumstance** — the time was never available at the right moment, and one clear afternoon closes it.
- **Design** — the ask is made at plan time, when engagement peaks, and collected after green, when the change feels finished and attention has moved on. On that reading it keeps not happening whatever the week.

The distinction matters because *"I have lots of work"* does not separate this adopter from any other. Every engineer who adopts this method will also have lots of work; that is the normal operating condition, not an exception, so a guard that only runs for people with spare attention is a guard that does not run. **One execution settles it.** Twenty minutes means circumstance and the guard is sound. Two hours means the design is asking for more than it is worth, and the method should shrink the ask, move the collection earlier, or stop listing it as a live mitigation.
**Independent of which is true:** the guard has not been exercised, so weakness #3's practical coverage is thinner than the rationale's mitigation list implies, and the document should say so rather than listing an unexercised guard beside exercised ones.
**A likely mechanism, worth testing before redesigning:** the hold-out is **requested at plan time but collected after green**. Plan time is peak engagement — the human is reading, answering blocking questions, making decisions. After green the change feels finished, the interesting part is over, and attention has already moved to the next thing. The method asks at the moment of highest attention and collects at the moment of lowest. Any fix should probably move the writing forward, not exhort harder about the collecting.
**Consequences already taken:** the v0.12.2 middle guard (human-verified expected values) was queued for reverting on the argument that a cheap fallback makes declining the expensive guard easier. That argument dies against a 0% baseline — there is no compliance for the fallback to erode, and a guard that runs beats a guard that does not. Kept.
**What this does to the grade:** the provisional A− was held provisional partly on the hold-out being pending. It is now clear the hold-out is not merely pending but structurally unexecuted, so weakness #3's practical coverage is thinner than the mitigation list implies. The rationale should say so rather than listing an unexercised guard beside exercised ones.
**Open, deliberately not designed tonight:** whether to make the ask dramatically smaller (one assertion, one number, written during the plan review), move it to plan time, or stop claiming it as a live mitigation and downgrade #3's coverage honestly. All three are real options; picking one on no sleep would repeat the mistake this finding is about.

### Finding #31 — the bundled scripts were unreachable for anyone who installed the plugin normally
**Observed (second cold review, verified against the official plugin reference before acting):** every script invocation in the skills used a *project-relative* path — `python3 scripts/check-plan.py`. For a marketplace install the working directory is the user's project while the scripts live in the plugin cache, so the deterministic layer simply was not there. Worse, a project that happens to have its own `scripts/check-plan.py` would have had *that* file executed instead.
**Why it went unnoticed for the whole pilot:** the author runs from a local clone with the plugin repo on disk, and the agent improvised absolute paths to it every session (visible in the transcripts as `/c/_Git/ctdd/scripts/...`). It worked, so nothing looked broken — the environment that made it work is the one environment no other user has. That is the same shape as finding #2 (Windows launcher) and #21 (uncaptured pin evidence): the mechanism appeared to exist, and the appearance was local to the author's setup.
**Fixed:** all script and reference paths now use `${CLAUDE_PLUGIN_ROOT}`, verified against the plugin reference as substituting *anywhere it appears in skill content*, and quoted because an install path can contain spaces.
**The review's own error, caught by verification:** it recommended `${CLAUDE_SKILL_DIR}` for files beside a skill. **No such variable exists** — the reference documents exactly three (`CLAUDE_PLUGIN_ROOT`, `CLAUDE_PLUGIN_DATA`, `CLAUDE_PROJECT_DIR`). Adopting that recommendation verbatim would have replaced a wrong-but-working relative path with a placeholder that never resolves, breaking the ADR template reference outright. The template is now addressed through the plugin root instead. This is the clearest case yet for the standing rule that an outside review is a *source*, not an authority: its diagnosis was right, its prescription was wrong, and only reproduction told them apart.

### Finding #32 — the working-tree check ran after plan approval, so three changes were planned against a baseline nobody had looked at
**Observed across three separate sessions:** the tree check lived in step 7, *after* the gate. On the generated-mapper change the agent added a package, then discovered the entire feature was uncommitted on `main`. On the query-object change it proposed a design, then found the human had already built a different abstraction solving an adjacent problem, and had to back out of its own approved plan — *"those solve different problems and would collide if I plow ahead with mine."* On the temp-doc change it found three uncommitted edits sitting on its own target file, mid-gate.
**Why after-the-gate is the wrong place:** what is already in the tree determines which tests get retrieved, what the agent believes current behaviour is, what contract delta it proposes, and what its file inventory says. All of that is settled during steps 2–5 and frozen by approval at step 6. Checking at step 7 confirms the baseline *after* every decision that depended on it.
**The distinction the old rule could not make:** this skill also triggers on "implement the reviewer comments on my PR," where an existing diff is the **input**, not contamination. The step-7 rule would have told the agent to ask for it to be stashed, which is exactly wrong. Contamination is unrelated local work or someone else's half-finished edits on the target files; an intentional PR diff is the thing being extended.
**Fixed (v0.13.3):** a step 0 that establishes branch, target, and staged/unstaged/untracked state before any retrieval, separates in-review work from contamination, and fixes the baseline every later diff check measures from. Step 7 keeps a re-check, since a tree can move while a plan is under review (which is what happened on the query-object change). The surface-check command now takes that baseline rather than always `HEAD` — `HEAD` alone misses everything already committed on a branch, so a PR-shaped change reported no surface at all.

### Finding #33 — the reference split silently swallowed four workflow rules, and a regex lookahead was the whole cause
**Observed (third cold review, reproduced immediately):** the v0.14.0 split moved step 10 into `references/colocated-notes.md` using a regex that ended at the next `## ` heading. Four sections that followed it were `###`, which that lookahead does not match, so **Bug fixes**, **Amendments**, **When artifacts disagree**, and **Standalone ADR requests** were carried into a file loaded only when a colocated note is being written. **Fixed:** the three workflow rules are back in the skill; standalone-ADR *routing* stays in the skill with its procedure in `adr-rules.md`; `colocated-notes.md` is now note craft only (1,328 → 630 tokens).

The 5.0k figure was therefore partly an artifact of the bug — some of the "saving" was rules going missing. The remaining legitimate move is splitting step 6's presentation and storage detail into a `plan-mode.md` reference, which is also the coherence fix a reviewer has now raised twice. Both are deferred together, because doing that restructure badly is what produced this finding.
**Second defect from the same review, also fixed:** `check-spec-surface.py --git` ran a bare `git diff` and so omitted untracked files — meaning the script's own documented convenience mode reopened exactly the new-test blind spot the skill's pipeline had just closed. A user following the simpler invocation would have silently lost the coverage. Now includes untracked files, with a regression test in a throwaway repo. Suite 101 → 102.
**Rejected from the same review, for the second time:** replacing "no status claim without a run in this turn" with "a prior run stays valid if nothing relevant changed." The proposed rule reinstates precisely the self-assessment that drifted three times — the agent deciding for itself that nothing relevant changed. The bright line is the mechanism, and the objection to it is comfort, not correctness.
**Also rejected, for the second time:** `${CLAUDE_SKILL_DIR}` as a path mechanism. It does not exist; the plugin reference documents three variables and that is not one of them. Two consecutive reviews have recommended it, which is a useful reminder that a confident reviewer repeating a claim does not make it true.

### Finding #34 — the surface checker under-reported in two more ways, and the tests written to prevent that were themselves being skipped
**Observed (fourth cold review, both claims reproduced before acting):**
- **A staged test change read as no surface at all.** `--git` ran a bare `git diff`, which compares the working tree against the *index* — so a test that had been modified and then `git add`ed reported `no test/contract/ADR surface touched`, exit 0. - **Untracked discovery depended on the current directory.** `git ls-files --others` is relative to cwd, so after any `cd` into a subdirectory a new test elsewhere in the repo vanished from the inventory. Reproduced: from `src/`, a new file under `tests/` was invisible. The agent changes directory constantly, so this was live. Both the script and the skill pipeline are now anchored to `${CLAUDE_PROJECT_DIR}` (a documented variable, unlike the one two earlier reviews recommended). A failing `ls-files` is now loud rather than silently empty.
**The sharper finding underneath:** the two regression tests written earlier the same day — the golden example test and the `--git` untracked test — sat *after* their file's `if __name__ == "__main__"` block, so each file's own advertised invocation silently skipped them. The tests added to prevent silent failure were themselves silently absent. Only the pytest command found them. Every `__main__` block now sits at the end of its file; direct execution went 11 → 15 and 29 → 32 tests.
**Built, because this exact defect shipped once already:** a structure test asserting that the four workflow routing sections stay in the always-loaded skill, that the notes reference holds only note craft, and that every `${CLAUDE_PLUGIN_ROOT}` path the skill mentions resolves to a real bundled file. The changelog for v0.14.1 said "nothing checks which rules survive a restructure." Now something does.

Every one reported success for the subset it managed to see. A checker's default failure mode is silence, and silence looks exactly like a pass. That is an argument for the machine-readable plan list already filed, and for testing checkers against *deliberately incomplete* inputs rather than well-formed ones.
**Three items from that review were skipped on the first pass, and the two that mattered were the regression tests for the very defects being fixed.** The bugs were corrected and shipped with no test written first — in a project whose own rule is that a test which has never failed is unvalidated as a detector, and after a full day of holding real changes to exactly that standard. Added in v0.14.3 and verified red-then-green against the previous code, which is the check that separates a detector from decoration. The third item, rooting plan and evidence paths at the project directory, was the same cwd defect in a different place and is fixed too. Worth recording plainly: the discipline is easiest to skip on your own work, and nothing in the workflow was watching me.

**Deferred again, now raised three times:** splitting step 6's presentation and storage protocol into its own reference. It remains ~705 words in a single paragraph, and moving it intact would not fix the incoherence — the lifecycle needs to become explicit states. That is a deliberate pass, not a fifth patch.

### Finding #35 — a fix broke CI for every conformant change, because one regex could not tell a file path from repository metadata
**Observed (fifth cold review, reproduced):** rooting plan paths at `${CLAUDE_PROJECT_DIR}` in v0.14.3 also rewrote the `CTDD-Plan:` pointer the skill puts in the MR description. After substitution that becomes an absolute path, and `check-plan.py --from-description` **rejects absolute pointers on purpose** — the description is untrusted CI input, and that sanitization was added deliberately in v0.9.6. Reproduced: the absolute form exits 1, the relative form validates and exits 0. So the previous release would have red-failed CI on every plan-carrying change.
**The distinction the fix missed:** filesystem operations want the project root, because a `cd` misplaces a relative write. Repository metadata wants to stay relative, because CI reads it in a different checkout at a different path, and an absolute path from an untrusted description is a security decision already made. One regex treated both as the same string.
**This is the second time a broad regex has done damage in this project**, after the split that swallowed four workflow sections into a conditional reference. Both were bulk transforms applied to prose without checking what else matched. The lesson is the one already learned on the anonymisation sweep and then not applied: the replacement is the easy part, and what breaks is everything adjacent that happened to match.
**Three fail-open paths hardened in the same release, all the same shape as the five already found:**
- `check-spec-surface.py` silently skipped input lines it could not parse and then printed "no test/contract/ADR surface touched" — a verdict it had not established, over input it had discarded. Malformed lines now report and exit 2 with no verdict.
- `check-plan.py` returned 0 when its triviality cross-check could not load, so CI could request a deterministic check, not get it, and pass anyway. Now exits 2: a trivial claim that was never verified is not a passing claim.
- `ctdd-review`'s prose described the corrected baseline handling while its executable example still showed the old unrooted, baseline-less command. Models copy examples, so the example was the operative instruction. It now calls `--git <baseline>`, which is the same defect as the plan example that a review caught two rounds ago.
**Running count of this shape: eight.** Extraction dropping names, extraction reading wrong sections, pin verification unrunnable, `--git` missing untracked, `--git` missing staged, unparseable lines skipped, cross-check failing open, and the example contradicting the prose. Every one reported success for the subset it could see. **A checker's default failure mode is silence, and silence is indistinguishable from a pass** — which is now the single most repeated lesson in this log, and an argument for writing checker tests against deliberately broken input first.
**All fixes verified red-then-green** against the pre-fix code, including the MR-pointer structure test, which is the discipline that was skipped one release earlier.

### Finding #36 — the fail-silent fix was applied at one boundary and not the composed one, which is the ninth instance of the same shape
**Observed (sixth cold review, reproduced):** v0.14.4 made `check-spec-surface.py` refuse a verdict over unparseable input. But `check-plan.py` *imports the same parser* and never checked the result, so the composed path still printed **"trivial claim stands — the diff touches no test/contract surface"** and exited 0 over input it could not read. A trivial claim reached CI unverified, which is precisely the gate that mode exists to be.
**Why the incomplete fix is the interesting part.** The defect was fixed where it was *found* — at the standalone boundary — rather than everywhere the parser is used. One release earlier the same thing happened with `--git`: untracked files fixed, staged files missed. The pattern behind the pattern is that a fail-silent defect is fixed at the call site that surfaced it, and the other call sites keep the old behaviour because nothing enumerates them. The check that would have caught both is trivial and was not done: *grep for every caller before declaring a fail-silent path closed.*
**Design fault removed rather than patched.** Malformed lines were being accumulated in a module-level list. Two callers shared it, nothing reset it between runs, and a second call could inherit the first's leftovers. `parse_name_status` now returns `(entries, malformed)`, so a caller cannot forget to clear shared state and cannot silently inherit another's. Both call sites now fail closed, with a regression test for each and one asserting the parser no longer leaks state between calls.
**Running count of the fail-silent shape: nine.** Extraction dropping names; extraction reading the wrong sections; pin verification unrunnable; `--git` missing untracked; `--git` missing staged; unparseable lines skipped; the triviality cross-check failing open; an example contradicting its own prose; and now the composed parser inheriting none of its sibling's caution.
**Also fixed, after three consecutive reviews raised it:** `ctdd-review` gated the hold-out question on *high-risk*, while the method's own worked example is `Risk: normal` with a hold-out required for money-path semantics. Risk describes implementation complexity; **load-bearing** describes the consequence of getting the semantics wrong, and they are different axes. A normal-risk payment amendment could have passed review with no hold-out question asked. The review skill now keys on load-bearing and says explicitly that it is not the same axis as the risk level.

### Finding #37 — the compaction limit is real, and measuring it inverted the fix everyone had assumed
**Verified rather than accepted:** four consecutive reviews cited a 5,000-token post-compaction limit and none of them was checked. The official documentation confirms it: when a conversation is summarized to free context, Claude Code re-attaches the most recent invocation of each skill **keeping only the first 5,000 tokens of each**, within a combined 25,000-token budget across skills. So the constraint is a runtime behaviour, not a style preference, and it has an ordering consequence nobody had drawn.
**What measuring it showed:** at ~5.9k tokens the skill loses ~950 tokens after compaction, and the assumption everyone had made — that the fix is splitting step 6 — was **wrong**. Every step-6 gate rule already survived, because steps sit early. What fell past the cliff was the **Guardrails** section, sitting last: *no status claim without a run* (the rule that replaced findings #8, #12 and #26), the preservation-detector rule, and the distributed-systems escalation. The rules most likely to be needed late in a long session were the ones guaranteed to be gone by then.
**The principle, which the documentation states directly and this skill was violating:** *write guidance that should apply throughout a task as standing instructions rather than one-time steps.* Guardrails apply throughout and were last. Sequential steps apply once and were first. Truncation keeps the front, so the file was ordered exactly backwards for the one thing that happens to long sessions.
**Fixed by reordering, not by cutting.** Guardrails now precede the workflow. Three more blocks moved up with them once it became clear they were never steps at all but condition-triggered rules: **amendments** (fires whenever a change touches an existing test — the modal case), **artifact conflicts** (a stop condition that can fire at any point), and the **bug-fix lane** (a classification rule). Each was truncated in an intermediate arrangement and each is now inside the surviving window. What is truncated instead: the standalone-ADR routing, the plan-format skeleton, and the ADR rules — every one of which has a reference the skill loads, and the plan format additionally has `check-plan.py` failing loudly when it is not followed.
**Built: a survival guard, which is a stronger check than the size guard everyone proposed.** `test_load_bearing_rules_survive_post_compaction_truncation` asserts that nine named rules fall within the first 5,000 tokens, not merely somewhere in the file. Verified by pushing one past the boundary, where it fails. Presence in a file is not the property that matters; presence in the surviving window is.
**Worth noting against six reviews:** every reviewer treated the limit as an argument for making the skill *smaller*. Measuring it showed the first-order fix is making it *better ordered* — the same total size, with the rules that must hold at every step stated before the steps. Splitting step 6 remains worth doing and is still guarded, but it was never the load-bearing repair.

### Finding #38 — the reference fallback did not survive the truncation it was supposed to survive
**Observed (seventh cold review, verified by measuring):** finding #37 justified truncating the plan skeleton, the ADR rules and the standalone-ADR routing on the grounds that each is "backed by a reference the skill loads." The reviewer checked where those *load instructions* actually sat: characters 21,364–23,793, i.e. **past the boundary themselves**. So the failure mode was worse than the one being solved — after compaction the agent would reach "assemble the plan (format below)" with the format gone *and* the instruction to fetch it gone, and `check-plan.py` can restore missing headings but not the plan's decision-summary craft, changed-assertion encoding, or hold-out guidance.
**The general rule this exposes:** when content moves to a reference, the *instruction to read it* becomes the load-bearing part, and it belongs where the action is triggered rather than beside the content it replaces. Loaders now sit inside the steps that need them — ADR rules at step 4, plan format at step 6 — and the notes loader sits in standing guidance, since a note can be warranted at any point. They now fall at characters 1,873–10,696.
**The test overstated what it proved, which is the more embarrassing half.** It sliced `[:5000 * 4]` and was named for surviving the 5,000-*token* limit — but 4 chars/token is an English-prose average and markdown with backticks, paths and code tokenizes worse, so the check proved "inside 20,000 characters" while claiming "inside 5,000 tokens." Renamed to name its own approximation, and tightened to a pessimistic 3 chars/token, because **the property worth asserting is margin, not a simulated tokenizer**. That tightening immediately failed, which is the test doing its job: the working-tree re-check sat at character 15,051, right on the boundary. It is now standing guidance (a tree moves *during* long sessions, which is exactly when compaction fires), and the furthest load-bearing rule sits at ~4,100 tokens even under the pessimistic proxy.
**A second test was also promising more than it asserted.** `test_every_reference_is_loaded_somewhere_before_it_is_needed` only checked that the filename appeared *somewhere*. It now asserts the loader precedes the inline section it backs — the property that matters, since that section is the part which gets truncated. My first attempt at that assertion was wrong in the opposite direction (it required the loader within 24 characters of the step heading, while it legitimately sits inside the step's opening sentence), which is worth recording: an over-strict guard is a false alarm that teaches people to ignore guards.
**Also fixed:** step 8 pointed at the "Amendments rule **below**" after the reorder moved it above — positional words break on every restructure, so it now just names the rule. And the plan skeleton offered `trivial` as a risk level while the authoritative format correctly forbids it, since a trivial change produces no plan at all; a guard now binds the two.

### Finding #39 — five critical fail-silent defects, including one that would have hit this codebase specifically
**Eighth cold review, all five reproduced before acting.** Every one is the same shape the log has now recorded ten times: the checker reported success over input it could not see.
- **Git-quoted non-ASCII paths defeated the trivial gate.** With `core.quotePath` at its default, git emits `M\t"tests/Ra\304\215unTests.cs"` for a path containing a c-caron, and the leading quote defeats every path pattern. A modified test file classified as untouched surface, and `check-plan.py` then let the change through as trivial — **the single rule the whole deterministic layer exists to enforce**. Reproduced, exit 0. This is not exotic for a Slovenian, German, French or Japanese codebase, and the pilot's own transcripts are full of Slovenian filenames. Paths are now unquoted in the parser, since a piped caller cannot set git's flags.
- **The step-9 pipeline reported a clean pass when git failed.** A bad baseline sent the error to stderr, left stdout empty, and the checker concluded "no surface touched", exit 0, with a modified test sitting in the tree. Empty stdin behaved identically. This is **finding #36's lesson unapplied**: `ctdd-review` had been switched to the returncode-checked `--git` invocation and `ctdd-change` step 9 plus `plan-format.md` still carried the raw shell pipeline. Both now use `--git`, and empty input refuses a verdict unless `--allow-empty` is passed.
- **Three more extraction shapes, the tenth instance.** A single explanatory sentence inside a test list terminated the section and the names after it were dropped; bold, italic and colon-separated bullets were dropped (four of five names in one measured plan); and a bullet whose text began with a section phrase *inverted* the classification rather than merely losing it. Hardened: emphasis is stripped, `:` is accepted as a separator, and a section now changes only on a label-shaped line rather than on any prose. **The real fix remains the machine-readable test list filed at #29** — ten instances is a design verdict on reading identifiers out of free-form markdown, and interim hardening narrows the surface without closing it.
- **The pin lane was unreachable for a pure-preservation refactor.** The format mandated both headings only "when both kinds are present", so a refactor that legitimately writes one heading got a false blocker in the default lane and a bare usage error in the pin lane — no correct path existed. Both headings are now unconditional (`Preservation pins: none` when empty) and the pin lane names the missing section instead of falling through.
- **The authz gate passed over a contract it could not read.** A `$ref`-composed OpenAPI document — the normal layout at any size — produced a zero-row matrix that `--check` then reported as *current*, exit 0, while the README claims that mode fails CI when an endpoint ships without rows. Now exits 2 on any skipped path item or a zero-operation check.
**A defect in my own fix, caught only because I ran it.** The first version of the authz guard read `if not operations:` — but `operations` is a *function* in that module, so the condition was always false and the guard could never fire. It would have shipped as "fixed". A guard that cannot fail is the fail-silent shape reproducing itself inside its own remedy, which is the most direct evidence yet that reasoning about a check is no substitute for executing it.
**Running count: fourteen.** Every one reported success for the subset it could see.

### Observation — ceremony cost, first real data point
**Observed:** the CTDD run took ~16 min of exploration-agent time plus multiple advisor passes; the no-skill plan was tighter and faster. On *this* load-bearing change the overhead earned its keep — CTDD caught the not-found-vs-empty hold-out, required the schema ADR, and pinned the deposited-status gap as a `currently_*` characterization test rather than shipping unfiltered behavior as if intended (all things the no-skill plan either made as silent decisions or left unmarked). Need the counterfactual — how it feels on the 10th routine change in a day — before drawing the curve. Keep timing each change and noting whether the gate caught anything that manual review would have missed.
